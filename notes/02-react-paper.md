## 0. 论文定位与核心问题

ReAct 是 Agent 推理策略的奠基论文（ICLR 2023）。它要解决的问题非常具体：

> 大语言模型在复杂任务中，要么只会"想"（CoT）——容易产生幻觉；要么只会"做"（Act-only）——缺乏高层规划。**能不能让模型交替进行推理和行动，形成闭环？**

论文的核心贡献不是提出一个新的模型架构，而是**提出一种动作空间的扩展方式**——把"思考"（Thought）当作一种特殊的、不改变外部环境的"动作"，插入到决策过程中。这个设计看似简单，但它统一了推理和行动，奠定了后续几乎所有 Agent 框架的基础。

---

## 1. 动机：为什么需要 ReAct？

### 1.1 纯推理的困境（CoT 的问题）

Chain-of-Thought 让模型逐步推理，在算术、常识、符号推理上表现突出。但它有一个结构性缺陷：

- **推理是静态的**——完全依赖模型内部知识，无法主动获取外部信息
- **错误传播**——某一步推理出错，后续步骤无法自我纠正（因为模型不知道哪步错了）
- **事实幻觉**——推理链看起来很流畅，但关键事实是编的

论文图 1 的 HotpotQA 例子直接展示了这一点：CoT 的推理看起来合理，但最终答案来自内部臆测，而非外部证据。后文表 2 量化了这个现象——CoT 失败案例中 **56% 是幻觉**。

### 1.2 纯行动的困境（Act-only 的问题）

另一类工作让模型在环境中执行动作（文本游戏、网页导航、API 调用），依赖观察直接输出下一步动作。问题是：

- 缺少高层规划——不知道当前动作服务于什么子目标
- 缺乏"工作记忆"——不维护对任务进度的显式追踪
- 遇到异常时容易重复无效动作或迷失方向

ALFWorld 的例子很直观：Act-only 找到了 knife，却不知道下一步该去 sinkbasin 清洗，在原地反复尝试。

### 1.3 ReAct 的解决方案

论文的洞察是：推理和行动不是替代关系，而是**互补关系**。

| 方向 | 机制 | 解决的问题 |
|---|---|---|
| **Reason to Act** | 推理指导行动 | 分解目标、制定计划、跟踪进度、处理异常 |
| **Act to Reason** | 行动辅助推理 | 检索外部知识、获取真实状态、验证假设、弥补内部知识缺口 |

这就是 "ReAct" 的含义：**Reason + Act**，两者交替形成闭环。

```
Thought → Action → Observation → Thought → Action → Observation → ...
```

## 2. 方法：ReAct 的形式化定义

### 2.1 动作空间的扩展

ReAct 在传统 RL 的动作空间上做了一个关键扩展：

$$
\hat{A} = A \cup L
$$

- **A**：环境中的真实动作（会改变环境、返回 Observation）
- **L**：语言空间中的"思维动作"（不改变环境、不返回 Observation，但更新内部上下文）

两类"动作"的根本区别：

| | 外部动作 $a_t \in A$ | 语言思维 $\hat{a}_t \in L$ |
|---|---|---|
| 改变外部环境？ | 是 | 否 |
| 返回 Observation？ | 是 | 否 |
| 影响后续决策？ | 通过 Observation 间接影响 | 直接更新上下文，作为后续动作的"工作记忆" |

这个设计的关键洞察：**Thought 不是事后解释，而是在决策过程中实时插入的上下文更新操作**。它让 LLM 能在每一步行动前显式地维护一个工作记忆。

### 2.2 Thought 的功能谱系

论文列举了 Thought 承担的多种功能，这不是泛泛的分类——不同功能对应不同的任务需求：

| 功能类别 | 具体表现 | 典型任务场景 |
|---|---|---|
| **任务分解** | "先找 x，再找 y，最后确定 z" | 多跳问答、长程决策 |
| **信息抽取** | 从 Observation 中提取关键事实 | 搜索后筛选相关内容 |
| **常识注入** | "刀更可能出现在抽屉或柜台" | ALFWorld 物体搜索 |
| **搜索改写** | 换一个关键词重新搜索 | 首次搜索无结果 |
| **进度跟踪** | "已拿到 knife，下一步清洗" | 多步家庭任务 |
| **答案综合** | 汇总多次检索结果，形成最终回答 | HotpotQA / FEVER |
| **异常处理** | "这个搜索方向不对，换一个策略" | 所有任务 |

**这解释了为什么 Inner Monologue 风格（只陈述目标和子目标）效果不如 ReAct**——它缺少了分解、推断、搜索改写、常识注入等灵活的推理功能。

### 2.3 实现方式：Few-Shot Prompting

论文的核心实验**不训练新模型**，而是：

- 使用冻结的 **PaLM-540B**
- 通过 **few-shot in-context prompting** 让模型模仿 ReAct 格式

Prompt 中放入人工书写的示例轨迹，每个轨迹包含 Thought → Action → Observation 的交替序列。人工示例很容易写——只需把人类在执行动作时的思考也写出来。

### 2.4 稀疏 vs 密集 Thought

论文在不同任务中使用不同密度的 Thought，这是一个重要的工程选择：

| 策略 | 适用任务 | 原因 |
|---|---|---|
| **密集 Thought**（几乎每步前都有） | HotpotQA、FEVER | 问题求解依赖显式多跳推理，每步检索前都需要明确"为什么搜这个" |
| **稀疏 Thought**（只在关键节点） | ALFWorld、WebShop | 动作序列可能很长（50+ 步），不需要每一步都"说话"，否则 prompt 爆炸 |

这个差异表明：**ReAct 不是一种固定的输出格式，而是一种灵活的决策框架**——Thought 的密度和类型应该根据任务特征调整。

## 3. 实验一：知识密集型推理任务

### 3.1 实验设计

**任务**：HotpotQA（多跳问答）、FEVER（事实验证）

**关键设定**：模型一开始**没有**支持文档，只能通过 Wikipedia API 获取外部信息。

**设计的 API**（故意弱于现代检索器，迫使模型显式推理检索策略）：

| 动作 | 功能 |
|---|---|
| `search[entity]` | 搜索实体，返回 Wikipedia 页面前 5 句 |
| `lookup[string]` | 在当前页面中查找包含某字符串的下一句（类似 Ctrl+F） |
| `finish[answer]` | 输出最终答案并结束 |

**基线方法**（基于同一 prompt 模板做消融）：
- **Standard**：去掉 Thought/Action/Observation，只保留问答格式
- **CoT**：去掉 Action/Observation，只留推理链
- **CoT-SC**：CoT + Self-Consistency（采样 21 条，多数投票）
- **Act**：去掉 Thought，只留 Action/Observation

### 3.2 核心结果

| 方法 | HotpotQA (EM) | FEVER (Acc) |
|---|---|---|
| Standard | 23.3 | 53.1 |
| Act | 25.7 | 58.9 |
| CoT | **29.4** | 56.3 |
| CoT-SC | 33.4 | 60.4 |
| ReAct | 27.4 | **60.9** |
| **ReAct → CoT-SC** (组合) | **35.1** | — |
| **CoT-SC → ReAct** (组合) | — | **64.6** |

几个关键观察：

**Act vs ReAct**：ReAct 在两个任务上都优于 Act，说明"只行动不思考"在知识任务中是不够的——检索需要推理来指导，答案需要推理来综合。

**ReAct vs CoT**：结论是**各有优劣，而非谁压倒谁**。
- HotpotQA 上 CoT 略高（29.4 vs 27.4）——因为有些问题的推理链不需要外部信息也能走通
- FEVER 上 ReAct 明显占优（60.9 vs 56.3）——因为事实验证对证据准确性的要求极高

**人工误差分析（表 2）揭示了更深的差异**：

| 失败模式 | CoT | ReAct |
|---|---|---|
| 幻觉（Hallucination） | **56%** | 极少 |
| 推理错误 | 较少 | **更多** |
| 搜索结果错误 | — | **23%** |
| False Positive（成功但实际有误） | **14%** | **6%** |

CoT 有时推理看起来顺，但事实是编的。ReAct 更 grounded（外部证据约束），但代价是推理灵活性受限——互锁的 Thought-Action-Observation 格式更拘束，可能导致搜索路线不佳、重复、陷入循环。另外，ReAct 受搜索质量制约严重：23% 的失败是因为检索不到有效页面。

> **个人理解**：这里有一个核心权衡——**grounding vs flexibility**。CoT 在内部知识的"自由联想"空间中探索，代价是失去事实约束；ReAct 被外部证据锚定，代价是检索失败时难以恢复。这不是 ReAct 的设计缺陷，而是两种认知策略固有的张力。后续所有的 Agent 架构（LangGraph、DSPy、Self-RAG）都在试图在这两极之间找到更好的平衡点。

### 3.3 ReAct 与 CoT 的互补性

论文最重要的发现之一：ReAct 和 CoT **不是替代关系，是互补关系**。两种组合策略都创造了最佳结果：

1. **ReAct → CoT-SC**：如果 ReAct 在限定步数内（HotpotQA 7 步 / FEVER 5 步）没出答案，退回 CoT-SC
2. **CoT-SC → ReAct**：如果 CoT-SC 的多数票不够稳定（< n/2），退回 ReAct 查外部知识

这说明最优策略不是"选 ReAct 还是选 CoT"，而是**根据内部知识自信度动态路由**——当代 Agentic RAG 中的 Self-RAG（自适应检索）思想，在这篇论文里已经有了雏形。

### 3.4 微调实验的关键启示

图 3 的结果对整个领域都很重要：

- Prompting 条件下，小模型学不好 ReAct
- 但用 **3000 条自动生成的正确轨迹微调后**：
  - PaLM-8B finetuned ReAct **超过所有 62B prompting 方法**
  - PaLM-62B finetuned ReAct **超过所有 540B prompting 方法**

作者的解释：微调 Standard/CoT 更多是在**记忆知识**（容易学到幻觉），微调 ReAct/Act 则是在**学习如何访问外部知识**——后者是一种更可泛化的能力。

> **个人理解**：这暗示了一个重要的方向判断——**Agent 能力的提升，未来的重心可能从"把模型训得更聪明"转向"让模型学会更好地跟外部世界交互"**。Tool Use 和检索策略的学习，比单纯的知识注入更有长远价值。

## 4. 实验二：交互决策任务

第四章把 ReAct 从知识推理扩展到长时程、稀疏奖励的交互决策任务：ALFWorld 和 WebShop。核心问题是：**语言推理能否显著改善长程行动？**

### 4.1 ALFWorld：长程家庭任务

ALFWorld 是一个文本版家庭环境（与 ALFRED 对齐），任务如"把干净的生菜放到桌子上"。难点：状态空间大、50+ 可交互位置、专家策略也需 50+ 步、奖励稀疏、需要常识推理。

**ReAct 使用稀疏 Thought**，主要功能：分解总目标、跟踪子目标完成、确定下一个子目标、利用常识推断物体位置。例如 prompt 中的 thought：

> "要完成任务，先找到 lettuce，再清洗，再放到 diningtable。lettuce 更可能出现在 fridge、diningtable、sinkbasin，先从 fridge 查起。"

**结果（表 3）**：

| 方法 | 成功率 |
|---|---|
| BUTLER (IL, best of 8) | 37% |
| Act (best of 6) | 45% |
| **ReAct (best of 6)** | **71%** |
| ReAct-IM (best of 6) | 53% |

- ReAct 比 Act 高 **26 个百分点**（相对提升 58%）
- 在 6 组受控 prompt 对比中，ReAct 全部优于 Act，相对提升 33%–90%

**Act-only 的典型失败**：在 knife 例子中，Act 找到 knife 后，没到 sinkbasin 就尝试清洗——因为它没有维护"我在哪个子目标"的追踪。ReAct 通过显式写出"现在已拿到 knife，下一步去 sinkbasin 清洗"避免了这种迷失。

### 4.2 WebShop：在线购物

WebShop 是真实网页购物环境（1.18M 商品，12k 人类指令），页面文本噪声大。用户要求如"带抽屉的床头柜，镍色，价格低于 $140"。

**ReAct 的 Thought 帮助模型完成"观察到决策"的桥接**——从噪声标题中提取属性、判断哪些选项与需求对应、决定继续搜索还是购买。

| 方法 | 得分 | 成功率 |
|---|---|---|
| IL | 59.9 | 29.1 |
| IL+RL | 62.4 | 28.7 |
| Act | 62.3 | 30.1 |
| **ReAct** | **66.6** | **40.0** |
| Human | 82.1 | 59.6 |

成功率从 30.1% 提升到 40.0%（+10 个百分点绝对值），且一次示例的 Act 就已匹敌 IL/IL+RL。

### 4.3 Inner Monologue 消融实验

作者设计了一个关键的消融：**ReAct-IM**，模仿 Huang et al. 的 Inner Monologue 风格。它只允许非常有限的思考——当前目标是什么、当前子目标是什么——但不包含子目标完成判断、下一步子目标确定、常识推断。

结果：ReAct = 71%，ReAct-IM = 53%。差距高达 18 个百分点。

这证明了一个很强的理论点：**ReAct 的优势不是"多说几句话"，而是"说对类型的思考"**。仅仅把环境反馈"语言化"（IM 风格）不够；真正有效的是高层分解、进度管理和常识注入——这些恰是 CoT/ToT 等纯推理方法擅长但被隔离在行动循环之外的能力。ReAct 的价值在于把它们带进了决策过程内部。

## 5. 关键设计决策与权衡

回顾全文，ReAct 的成功依赖于几个关键设计决策，每个都隐含着权衡：

| 设计决策 | 得 | 失 |
|---|---|---|
| 用 Prompting 而非训练 | 灵活、易调试、可解释、不需要训练数据 | 依赖 prompt 质量、小模型不适用 |
| Thought 不改变外部环境 | 安全、可回溯、不影响真实状态 | Thought 可能是错的而 Agent 不自知 |
| 受限的 Wikipedia API（弱于现代检索器） | 迫使显式推理检索策略，更接近人类行为 | 23% 失败来自检索不到页面——推理再好也救不了 |
| 文本格式而非结构化 JSON | 自然、灵活、易写示例 | 解析脆弱——你下周手写 Agent 时就会遇到 |
| 稀疏 vs 密集 Thought 按任务调整 | 灵活适应不同任务需求 | 没有统一的自动化决策规则——需要人工判断 |
| 限定最大步数 + fallback 策略 | 防止死循环 | 可能提前终止有价值的探索 |
